### 3) Синтез (карта применимости “область → skill”)

1. **Integrator CLI (ядро проекта)**  
   Использовать: `integrator-cli-engineer`, затем `github-pr-reviewer`, затем `github-security-reviewer`.  
   Обоснование: цепочка закрывает реализацию CLI, функциональный review и security review перед merge.

2. **LocalAI assistant (RAG/SSOT/indexing)**  
   Использовать: `localai-assistant-ops` + `security-ops`.  
   Обоснование: первый закрывает доменную эксплуатацию RAG и SSOT, второй фиксирует security baseline.

3. **Базы знаний (Agent Memory, Obsidian workflow)**  
   Использовать: `localai-assistant-ops` как основной маршрутизатор операций, `github-security-reviewer` для токенов/API, `github-pr-reviewer` для quality перед интеграцией.  
   Обоснование: знания связаны с RAG/memory pipeline и внешними API.

4. **Подпроекты VPN**  
   Использовать: `vpn-manager-maintainer` и `vpn-manager-fedora-maintainer` + `security-ops`.  
   Обоснование: профильные maintainer-skills уже существуют и закрывают release/config контур.

5. **Claude Stealth Connect подпроект**  
   Использовать: `claude-stealth-connect-ops` + `security-ops`.  
   Обоснование: отдельный домен VPS/proxy требует операционного и security контроля.

### 5) Ограничения и противоречия

- Противоречие стандартов размещения: текущий проектный путь `.trae/skills` и внешний стандарт `.agents/skills` требуют явной политики совместимости.  
  Статус: **Противоречиво**.  
  Факт: текущая структура и внешний стандарт различаются.

- Неполнота метаданных в текущих skill-папках: не везде есть расширенные UI/policy/dependencies-метаданные через `agents/openai.yaml`.  
  Статус: **Проверено**.  
  Факт: в skill-деревьях `openai.yaml` не обнаружен.

- Не проверено фактическое поведение implicit invocation для каждого skill через набор тестовых промптов.  
  Статус: **Не проверено**.  
  Факт: в репозитории нет отчёта с матрицей trigger-тестов по каждому skill.

## Рекомендации

1. Создать `docs/SKILLS_INDEX.md` и зафиксировать реестр: `skill`, `scope`, `trigger`, `anti-trigger`, `owner`, `path`, `security gate`.  
   Аргумент: снижает стоимость роутинга и ошибки выбора skill, ускоряет работу по подпроектам и БЗ.

2. Ввести для каждого skill чёткую пару `когда включается / когда не включается` и сократить пересечения описаний.  
   Аргумент: повышает точность implicit invocation и уменьшает ложные срабатывания.

3. Добавить `agents/openai.yaml` в ключевые skills (`integrator-cli-engineer`, `localai-assistant-ops`, `security-ops`, `github-security-reviewer`) с `allow_implicit_invocation`, `display_name`, `dependencies.tools`.  
   Аргумент: улучшает управляемость, UX и предсказуемость использования.

4. Зафиксировать единый compatibility-слой по размещению skills: поддерживать текущую `.trae/skills` структуру и добавить зеркальный read-only индекс под стандартный путь, чтобы не терять совместимость с внешними экосистемами.  
   Аргумент: снимает риск vendor lock-in и упрощает переносимость.

5. Включить обязательный quality-поток для изменений skills: `github-pr-reviewer` + `github-security-reviewer` + `security-ops` в одном чек-листе перед merge.  
   Аргумент: стабилизирует качество и снижает операционные риски.

6. Создать отдельный skill для governance баз знаний (Agent Memory + Obsidian + GitHub Issues memory-loop) и сделать его официальным маршрутизатором знаний.  
   Аргумент: закрывает выявленный пробел, убирает размытую ответственность, ускоряет задачи по БЗ.

7. Провести trigger-validation матрицу для всех 17 skills и сохранить отчёт в `reports/` с примерами входных задач и ожидаемого выбора skill.  
   Аргумент: переводит выбор skills из интуитивного режима в проверяемый контракт.
