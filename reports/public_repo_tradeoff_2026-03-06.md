# Public repo trade-off (2026-03-06)

## Коротко
- Если сделать репозиторий публичным, можно включить branch protection без GitHub Pro.
- Это разблокирует `PROC-1` и `P1-4` (обязательные проверки и минимум 2 апрува до merge в `main`).

## Что нужно изменить при переводе в public
- Проверить историю и текущие файлы на секреты (токены, ключи, приватные URL, внутренние пути).
- Подтвердить, что `vault/` и локальные `.env` не попадают в репозиторий.
- Обновить README: убрать приватные организационные детали, оставить публично безопасную документацию.
- Включить branch protection сразу после перевода в public:
  - required check: `ci / test`
  - required approvals: `2`
  - dismiss stale approvals: `on`

## Что станет лучше
- Бесплатная разблокировка технического merge-замка для `main`.
- Прозрачность разработки и легче получать внешний фидбек.
- Проще показывать прогресс и качество (CI, отчёты, тесты) без ручного объяснения.

## Какие риски вырастут
- Утечка чувствительных данных, если в истории остались секреты.
- Увеличение внешнего шума: спам-issue/PR, боты, нежелательные обсуждения.
- Публичная видимость архитектуры и рабочих процессов.

## Как уменьшить риски
- Перед переводом сделать security-pass: gitleaks + pip-audit + ручной просмотр конфигов.
- Включить CODEOWNERS и шаблоны issue/PR для фильтрации шума.
- Не хранить токены в репозитории вообще; только через локальные секреты и env.

## Решение на сейчас
- GitHub Pro исключён до появления дохода.
- Практический путь: подготовка security-перевода в public, затем закрытие `PROC-1` и `P1-4`.
