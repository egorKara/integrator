# Оценка бесплатных AI-сервисов для 24/7 комбайна (2026-03-05)

## Метод
- В shortlist включены только сервисы с официальной документацией по pricing/limits/policy.
- Неофициальные калькуляторы и форумы использованы только как сигналы, но не как SSOT.
- Критерии: стабильность лимитов, прозрачность billing, простота интеграции, риск lock-out, применимость к production-like 24/7.

## Shortlist (официально подтверждаемые)

### 1) OpenRouter
- Плюсы:
  - OpenAI-compatible интеграция.
  - Есть бесплатные модели и бесплатный план.
  - Есть авто-роутинг и fallback.
- Ограничения:
  - Free tier имеет жёсткие лимиты (req/day, rate limits), при пиках могут быть provider-side throttling.
  - “Бесплатные модели” могут меняться динамически.
- Роль в комбайне:
  - **Fallback B** после LM Studio, но не primary production SLA.

### 2) Hugging Face Inference Providers
- Плюсы:
  - Единый routing-layer, OpenAI-compatible для chat completion.
  - Прозрачная модель routed/custom provider key.
  - Есть ежемесячные стартовые credits.
- Ограничения:
  - Free credits небольшие, быстро исчерпываются.
  - Для устойчивого 24/7 может потребоваться платный режим.
- Роль в комбайне:
  - **Fallback C**/эксперименты; полезен как маршрутизатор к нескольким провайдерам.

### 3) Cloudflare Workers AI
- Плюсы:
  - Есть бесплатная ежедневная аллокация.
  - Хорош для always-on serverless gateway + observability (AI Gateway).
  - Удобен как “периметр” с rate limiting/caching.
- Ограничения:
  - Бесплатный бюджет лимитирован daily quota.
  - При росте нагрузки нужен paid Workers plan.
- Роль в комбайне:
  - **Infrastructure-level fallback/runtime gateway**, не “бесплатный бесконечный LLM”.

### 4) Groq (Free tier)
- Плюсы:
  - Очень быстрый inference для определённых моделей.
  - Есть free tier без карты (по официальным FAQ/limits docs).
- Ограничения:
  - Жёсткие org/model-level rate limits.
  - Free tier не рассчитан на высокую 24/7 нагрузку.
- Роль в комбайне:
  - **Burst fallback** для быстрых ответов, когда local degraded.

## Критический вывод по “бесплатности”
- “Полностью бесплатный 24/7 ассистент без деградации” практически недостижим.
- Реалистичный путь: **local-first + селективные free fallback + строгий бюджетный контроль + graceful degradation**.

## Неудобные, но насущные вопросы (по внешним free AI)
1. Что делаем при одновременном срабатывании rate-limit на двух fallback-провайдерах?
2. Какие данные запрещено отправлять во внешние API даже при аварии local-contour?
3. Кто owner решения “платим/не платим”, когда бесплатный режим перестаёт выдерживать нагрузку?
4. Какой допустимый уровень “ответил, но хуже качеством” в ночном 24/7 режиме?
5. Есть ли юридическое ограничение по отправке проектного кода в third-party провайдеры?

## Рейтинг пригодности (для текущих вводных)
- **A- (рекомендуемый fallback): OpenRouter**
- **B+ (маршрутизируемый fallback): Hugging Face Providers**
- **B (инфраструктурный runtime): Cloudflare Workers AI**
- **B (burst speed fallback): Groq free tier**

## Рекомендация
- Не строить архитектуру на одном free-provider.
- Использовать free-провайдеры только как **ограниченные fallback-слои** с явными circuit-breaker и лимитными квотами.
