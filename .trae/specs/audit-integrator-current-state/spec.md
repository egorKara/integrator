# Спецификация: Детальный аудит текущего состояния integrator

## Why
Проекту нужен воспроизводимый и доказательный срез текущего состояния без предположений и устаревших оценок.  
Это снижает риск неверных приоритетов и позволяет формировать бэклог на основе подтверждённых фактов.

## What Changes
- Выполнить детальный аудит репозитория `integrator` по фиксированным доменам: структура, документация, CI/CD, тесты, GitHub-статус, стек и зависимости, метрики производительности/доступности.
- Фиксировать в отчёте только подтверждённые факты с явной пометкой статуса (`Проверено` / `Не проверено` / `Противоречиво`) и ссылкой на артефакт проверки.
- Сформировать диалектический блок `тезис–антитезис–синтез` на базе актуального roadmap/todo и фактических результатов аудита.
- Сформировать приоритизированный бэклог действий с критичностью, оценкой трудозатрат и измеримым Definition of Done.
- Сохранить в репозитории два итоговых артефакта: структурированный аудит-отчёт и приоритизированный бэклог.

## Impact
- Affected specs: Аудит проекта, управление качеством, планирование roadmap, контроль CI/CD.
- Affected code: `reports/` (новые/обновлённые отчёты), команды проверки в CLI (`integrator`, `git`, локальные quality-команды), GitHub API-артефакты в рамках существующих инструментов проекта.

## ADDED Requirements
### Requirement: Доказательный аудит состояния проекта
Система процесса аудита SHALL собирать и публиковать только подтверждённые данные по заданным доменам состояния проекта.

#### Scenario: Успешный сбор фактов по всем доменам
- **WHEN** запускается аудит текущего состояния `integrator`
- **THEN** формируется структурированный отчёт с разделами по всем запрошенным доменам
- **AND** каждый пункт содержит статус достоверности (`Проверено` / `Не проверено` / `Противоречиво`)
- **AND** у каждого пункта указан факт подтверждения/опровержения (команда, файл, артефакт)

### Requirement: Диалектическая систематизация результатов
Система SHALL включать блок `тезис–антитезис–синтез`, связанный с roadmap/todo и фактическим состоянием.

#### Scenario: Успешное формирование приоритетов
- **WHEN** завершён фактический аудит
- **THEN** формируется тезис из заявленных целей roadmap/todo
- **AND** формируется антитезис из обнаруженных пробелов/противоречий
- **AND** формируется синтез как приоритизированный список действий (фичи, баги, рефакторинг, инфраструктура)
- **AND** для каждого действия указаны критичность, трудозатраты и измеримый Definition of Done

### Requirement: Сохранение артефактов в репозитории
Система SHALL сохранять итоговые артефакты аудита в репозитории проекта.

#### Scenario: Успешная публикация результатов
- **WHEN** аудит и бэклог завершены
- **THEN** в репозитории сохранён итоговый аудит-отчёт
- **AND** в репозитории сохранён приоритизированный бэклог
- **AND** оба документа доступны для последующего обновления и трассировки изменений

## MODIFIED Requirements
### Requirement: Формат аудиторских отчётов
Аудиторские отчёты проекта MUST использовать единый формат фиксации достоверности по каждому пункту (`Проверено` / `Не проверено` / `Противоречиво`) и содержать явное доказательство факта.

## REMOVED Requirements
### Requirement: Неформализованные выводы без верифицируемых артефактов
**Reason**: Необоснованные выводы и общие формулировки снижают точность приоритизации и мешают воспроизводимости аудита.  
**Migration**: Все новые и обновляемые аудиты оформляются в доказательном формате со статусом достоверности и подтверждающими артефактами.
