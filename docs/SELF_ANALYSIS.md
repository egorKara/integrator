# Самоанализ (канон)

## Назначение
Самоанализ — короткая рефлексия по итогам работ, которая помогает:
- закреплять успешные практики (что повторять);
- выявлять причины сбоев (что менять);
- снижать дрейф правил и “скрытую импровизацию”.

## Правила
- 1–2 экрана максимум: лаконично.
- Без секретов, без токенов, без приватных данных.
- Только то, что переносимо между сессиями.
- Привязка к evidence (команды/артефакты/PR) вместо общих слов.

## Шаблон (T+A=S)

### Тезис
- Что пытались сделать (цель), где (репо/проект), какой критерий “готово”.

### Антитезис
- Что мешало (неизвестные, ограничения, риски), что оказалось не так.

### Синтез
- Что сделали (решения), почему это работает, как проверить (evidence).

### Уроки (3–7 bullets)
- Что повторять.
- Что прекратить делать.
- Что автоматизировать.

### Next atomic step
- Один следующий шаг без ветвлений.

### Антитезис
- Fine-grained PAT даёт 403 на Actions secrets/workflows даже после изменения permissions; трудно отличить “не те права” от “не применились”.
- Риск утечки токена в логи при неосторожных командах.

### Синтез
- Применили протокол “token discovery → fact checks → mutations”:
  - discovery: проверка наличия токена только по факту (present/len), без вывода значения;
  - fact checks: проверка доступности `actions/secrets/public-key` перед `gh secret set`;
  - mutations: создание репо, push, установка secret, запуск workflow, проверка PR’ов.
- Для 403 на secrets/workflows: Regenerate PAT + обновление `GH_TOKEN` решают проблему быстрее, чем “угадывание” настроек.

- В PowerShell не использовать `<` для stdin; вместо этого `Get-Content -Raw | ...`.

## Самоанализ — 2026-03-02 (Trae terminal «с» spam)

### Тезис
- Цель: убрать самопроизвольный “ввод” `с`/`сclear` в терминале Trae; готово, когда в терминале нет ложных команд и красных ошибок “term is not recognized”.

### Антитезис
- Симптом похож на “залипание клавиши”, но по наблюдениям это не физический ввод; требовалось доказательство источника на уровне факта (какие именно строки попадают в shell).
- Часть событий происходила внутри IDE (инжект команд), а не в репозитории; изменения требовали аккуратности (без секретов и без разрушения окружения).

### Синтез
- Зафиксировали, что в PowerShell исполняются строки с кириллической `с` (`сclear`, `сRemove-Variable ...`) и служебные сообщения `[Trae] Shell integration ...`/`[Trae] Safe Rm alias ...`; это означает автосенд команд в терминал со стороны Trae.
- Root cause: цикл “переинжекции” терминальной интеграции (shellIntegration + safe_rm) в Trae и попадание кириллической `с` в начало инжектируемых команд (вероятно из подсистемы ввода/IME при активной русской раскладке).
- Митигация: отключить terminal shell integration в Trae и убрать SafeRm env для интегрированного терминала; в репо добавили инструменты для диагностики инжекта в keytrace.
- Evidence:
  - `%APPDATA%\Microsoft\Windows\PowerShell\PSReadLine\ConsoleHost_history.txt` содержит исполнявшиеся строки `с...` и `[Trae] ...`.
  - [trace-c.ps1](file:///c:/integrator/keytrace/trace-c.ps1) расширен режимом `-IncludeInjectedFields` для сбора `lower_il_injected` и `dwExtraInfo`.

### Уроки
- Отличать “вывод программы” от “ввода в shell”: проверка по PSReadLine history быстрее и надёжнее скриншотов.
- Для проблем ввода всегда проверять “латиница vs кириллица” (`c` vs `с`) как отдельный класс багов.
- При IDE-инжекте фиксировать root cause через артефакты вне репо (history/logs), а правки в репо держать как диагностику/обходные инструменты.
- Любые автопочинки терминала (shell integration) должны иметь безопасный kill-switch (настройка/флаг), иначе они становятся источником шума.

### Next atomic step
- После перезапуска Trae подтвердить отсутствие `[Trae] ...` и `с...` в терминале; если воспроизводится — оформить минимальный баг-репорт с выжимкой из PSReadLine history (без приватных данных).
