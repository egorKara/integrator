# Заповедник промптов (канон)

## Тезис
“Заповедник промптов” — это долговременная память о рабочих формулировках запросов, стратегиях и шаблонах диалога, которые дают воспроизводимый результат. Он нужен, чтобы не изобретать одно и то же при каждой сессии и чтобы улучшать качество выполнения задач через накопление практик.

## Антитезис (риски)
- Промпты без контекста быстро деградируют: “магия” перестаёт работать при изменении проекта/инструментов.
- В промпты и примеры легко случайно занести секреты, токены, приватные пути/данные.
- Заповедник превращается в “свалку”, если нет структуры (теги/шаблон/критерии качества).

## Синтез (как вести заповедник)

### Где хранится
- Оперативные записи сессий: `.trae/memory/Заповедник промтов - <YYYY-MM-DD-HHMM>.md`
- Текущий указатель: `.trae/memory/zapovednik_current.txt`
- Этот документ (`docs/ZAPOVEDNIK_PROMPTOV.md`) — канон: правила ведения + отобранные примеры.

### Правила (обязательные)
- **Без секретов**: никогда не сохранять токены/пароли/ключи/PII. Если пример требует “секрета” — использовать плейсхолдеры.
- **Evidence-based**: фиксировать, что именно было проверено (команда/лог/ссылка на артефакт).
- **T+A=S**: хороший промпт задаёт формат ответа и фиксирует Next atomic step.
- **Одна цель**: промпт должен иметь одну ясную цель и критерий успеха.

### Шаблон записи
Используй этот блок как стандарт:

```md
## Задача: <кратко>

### Контекст (минимум)
- repo/project:
- ограничения:
- что считается “готово”:

### Промпт (готовый к повторному использованию)
<текст промпта>

### Верификация (evidence)
- команды:
- артефакты (пути/ссылки):

### Итог и самоанализ (1–3 bullets)
- что сработало:
- что улучшить:
```

## Отобранные промпты (из .trae/memory)

### Intake по GitHub Issue (agent workflow)
**Промпт**
```md
Открой GitHub issue <URL или #N> и сделай intake по протоколу AGENT_TASK_INTAKE.md.

Требования:
- Выпиши T+A=S (1–3 bullets на секцию) строго фактами из issue/кода.
- Сформируй TodoWrite: 2–5 крупных задач, один in_progress.
- Выполни Next atomic step (один, без ветвлений).
- Проверь результат (ruff/mypy/unittest или релевантные проверки).
- Обнови issue: комментарий “сделано/осталось/verify/next step”.
- Секреты никогда не выводи; токен только token_present=true/false.
```

**Самоанализ**
- Лучше всего работает, когда “Next atomic step” действительно один и проверяемый.
- Улучшение: если issue не по шаблону, агент сначала добавляет комментарий с T+A=S и просит уточнить “готово/verify”.

### Комментарий в issue с доказательствами (verify)
**Промпт**
```md
Составь комментарий в GitHub issue по шаблону:
- Сделано:
- Осталось:
- Верификация (команды + результат):
- Артефакты (пути/ссылки):
- Next atomic step:

Ограничения:
- не включай секреты/токены/ключи;
- если есть внешние ссылки — только на публичные файлы/issue/PR.
```

### Аудит proxy/VPS цепочки и утечек
**Промпты**
- Поручить диагностику VPS/proxy цепочек с подтверждением фактами и логами.
- Проверить обходные пути на ноутбуке и приложить точные пути логов.

**Самоанализ**
- Лучше всего сработала комбинация: факты + точные пути логов + короткие определения терминов.
- Улучшение: явно разделять “ноутбук / клиент / VPS / proxy” и проверять маршруты по интерфейсам до выводов.

### Git hygiene (безопасно)
**Промпты**
- Провести гигиену репозитория, избегая опасных чисток.
- Зафиксировать текущее состояние отдельным отчётом.

**Самоанализ**
- Отчёт по `git status` полезнее “попыток угадать”, что можно удалить.
- Улучшение: на реальную чистку нужна явная команда на модификации.

### Запуск RAG и semantic apply (пример дисциплины)
**Промпты**
- Запустить RAG сервер в фоне и проверить доступность порта.
- Применить actions semantic на весь Vault через SSOT override.

**Самоанализ**
- Улучшение: SSOT override должен повторять paths/servers из SSOT, иначе дрейф.
