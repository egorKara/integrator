# Протокол: агент разбирает задачу из GitHub Issue

## Цель
- Дать воспроизводимый способ “подхватить” работу после паузы: issue = внешний контекст, project = доска внимания.
- Сделать работу агента проверяемой: каждый шаг заканчивается фактами (verify) и одним Next atomic step.

## Входные данные
- URL issue или номер issue в репозитории.
- Тело issue по шаблону “Memory Task (T+A=S)” и/или комментарии.

## Выходные данные (обязательные)
- Сводка T+A=S (1–3 bullets на секцию).
- Один Next atomic step без ветвлений.
- Доказательства верификации (команды/артефакты) без секретов.
- Обновление issue: комментарий с результатами и ссылками; закрытие только при Verification=Passing.

## Алгоритм intake (строго)
1) Прочитать issue целиком (body + последние 3–10 комментариев).
2) Выписать T+A=S:
   - Тезис: факты “что есть сейчас”.
   - Антитезис: риски/неизвестные/ограничения.
   - Синтез: решение/подход и “что считаем готово”.
3) Извлечь инварианты (что нельзя ломать):
   - безопасность (секреты/токены не логировать, не коммитить);
   - контракты (CLI JSON, формат артефактов, SSOT).
4) Перевести задачу в локальный список работ:
   - 2–5 крупных шагов (без микрошагов).
   - 1 шаг в progress одновременно.
5) Выполнить Next atomic step:
   - если неясно — собрать факты (поиск/чтение/диагностика), не гадать.
6) Проверить результат:
   - ruff/mypy/unittest или другой релевантный набор проверок;
   - smoke проверка сценария.
7) Записать back в issue:
   - “Что сделано” + “Что осталось” + ссылки на файлы/артефакты;
   - указать новый Next atomic step.
8) Закрывать issue только если:
   - Verification=Passing (или явное исключение с причиной);
   - SSOT обновлён, если затронут контракт/правило.

## Правила безопасности (не обсуждается)
- Никогда не вставлять токены/секреты в issue/PR/логи/скриншоты.
- В комментариях допускается только `token_present=true/false`, без значения.
